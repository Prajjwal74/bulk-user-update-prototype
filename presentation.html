Product Case Study

Bulk User Information
Update

A comprehensive solution for making attribute changes across hundreds of employees simultaneously — with approval workflows, scheduling, validation, and downstream system propagation.

Prototype Built Edge Cases Covered Data Propagation Addressed

Context

Problem Statement

Rippling administrators currently change employee attributes one at a time. Customers need the ability to make changes affecting dozens to hundreds of employees at once.

🏢 Scale

Must handle 12 to 500+ employees in a single bulk operation.

🔀 Multi-Attribute

Multiple fields changed simultaneously, with potentially different values per employee.

🔗 Interconnected

Changes propagate to first-party products (Payroll, Insurance) and third-party integrations (Slack, GitHub).

Key Constraints

  • Updates only — no bulk onboarding or terminations
  • Mixed value changes — e.g., different compensation per employee after performance reviews
  • Downstream impact — every change cascades across Rippling's ecosystem
  • Sensitive fields — compensation, currency, and status require approval workflows

Question 1

How Did I Choose My Approach?

Three primary approaches were evaluated based on user needs, scalability, and flexibility.

Option A

Spreadsheet-Only Upload

Users download a template CSV, make changes offline, and re-upload for processing.

✓ Pros
  • Familiar Excel workflow
  • Handles per-user different values naturally
  • Good for very large batches
✕ Cons
  • Error-prone (typos, format issues)
  • No real-time validation
  • Poor for uniform changes
Option B

Filter + Rule-Based Only

Select users via filters, then apply uniform operations (increase salary by 10%, change department, etc.).

✓ Pros
  • No file handling needed
  • Fast for uniform changes
  • In-app validation
✕ Cons
  • Cannot handle per-user values
  • Limited for complex scenarios
  • Doesn't scale for mixed changes
✓ Chosen

Hybrid: Filter + Rules & CSV Upload

Combine both methods — users choose the best tool for their specific use case within the same flow.

✓ Pros
  • Best of both worlds
  • Handles all use cases
  • Flexible for any batch size
  • In-app + file validation
✕ Cons
  • More complex to build
  • Two paths to test & maintain

Design Principles

Progressive Disclosure

Simple operations stay simple. Complexity is available when needed but never forced on the user.

Guardrails, Not Gatekeeping

Validation warns about conflicts (currency mismatches, cross-department changes) without hard-blocking unless necessary.

Auditability First

Every change is tracked with full history, downloadable change sheets, and reversal capabilities.

Approval Where It Matters

Only sensitive fields (compensation, currency, status) trigger approval. Non-sensitive changes apply directly.

Approval & Application Status

Conditional Approval Paths

🔒 Sensitive Fields Changed

Status: Pending Approval. Request sent to HR Admin. Admin can approve or reject. On approval, changes begin processing.

✅ No Sensitive Fields

Status: Processing. Changes apply directly with progress bar. No approval gate.

📅 Scheduled + Sensitive

Status: Scheduled — Pending Approval. Both the scheduled date and approval gate apply. Changes won't run on the scheduled date until approved.

📅 Scheduled + Non-Sensitive

Status: Scheduled. Changes will auto-apply on the scheduled date without approval.

Processing & Error States

⟳ Processing

Live progress bar showing % complete. Option to cancel mid-processing.

⚠ Partial Failure

~15% of users may fail (simulated). Shows "X of Y applied successfully". Options: Retry failed or Continue without failed.

✕ Rejected

Approver rejected the request. Option to Edit and resubmit.

⊘ Cancelled

Processing cancelled mid-way. No changes applied. Option to edit and resubmit.

Feature Deep-Dive

Scheduling & Approval Matrix

How Scheduling Works

In the confirmation modal, a 📅 Schedule button (inspired by Blinkit's delivery scheduling UX) lets users pick a future effective date. The scheduled date appears as a separate highlighted banner with a clear option.

UX Flow

  1. User clicks "Submit for approval" or "Confirm and apply"
  2. Confirmation modal opens with 📅 Schedule button in header row
  3. Clicking Schedule opens native date picker (min: today)
  4. Selected date shows as purple banner: "📅 Scheduled for Tue, Mar 4" with Clear option
  5. CTA remains contextual: "Submit for approval" (sensitive) or "Schedule changes" (non-sensitive)

Decision Matrix

Sensitive Fields?Scheduled?StatusWhat Happens
NoNoProcessingApplies immediately
NoYesScheduledAuto-applies on date
YesNoPendingNeeds approval first
YesYesSched + PendingNeeds approval, then applies on date

Key Principle

Scheduling never bypasses approval. If sensitive fields are being modified, approval is always required regardless of when the changes are scheduled to take effect. This ensures compliance and audit integrity.

Question 3

Challenges of Data
Propagation

How modified data flows from Rippling's system of record to first-party products and third-party integrations — and the challenges that arise at scale.

Question 3 — Challenges

Data Propagation Challenges

First-Party Systems

1

Atomicity & Consistency

When changing compensation for 200 employees, Payroll and Benefits must both receive the update. A partial sync (Payroll updated but not Benefits) creates data inconsistency and compliance risk.

2

Ordering & Dependencies

Status changes may need to propagate to Payroll before Device Management (to deactivate payroll before reclaiming devices). Bulk changes amplify ordering complexity.

3

Payroll Cut-Off Timing

Bulk compensation changes near payroll processing deadlines risk either double-payment or missed payments. The scheduling feature helps but doesn't eliminate the risk.

Third-Party Integrations

4

Rate Limiting & Throttling

Updating 500 users' titles on Slack, Google Workspace, and GitHub simultaneously means 1,500+ API calls. Third-party rate limits can cause sync delays and partial failures.

5

Eventual Consistency

Different third-party APIs have different propagation speeds. Slack may update in seconds while Google Workspace takes minutes. Users may see inconsistent data across platforms during the sync window.

6

API Failures & Rollbacks

A third-party API outage (e.g., GitHub is down) shouldn't block the entire bulk change. But partial propagation creates an inconsistent state that's hard to reconcile later.

Question 3 — Solutions

How I Would Address These Challenges

🔄 Async Job Queue with Retry

Each downstream sync is an independent job on a queue (e.g., SQS/Kafka). Failed jobs retry with exponential backoff. This decouples the core change from propagation and handles third-party rate limits gracefully.

📊 Per-System Sync Status Dashboard

Instead of a binary "applied/failed", show admins the sync status per downstream system: "Payroll ✅ · Benefits ✅ · Slack ⟳ · GitHub ❌ retry in 5m". This is already reflected in our "Affected Systems" preview.

🔀 Dependency Graph Execution

Define a DAG of system dependencies (e.g., Payroll → Benefits → Device Management). Process in topological order. If a parent fails, child syncs are held until the parent succeeds.

🛡️ Idempotent Operations

Every sync operation must be idempotent — safe to retry without side effects. Include a change-request-ID in every API call so downstream systems can deduplicate.

📅 Scheduling as a Buffer

The scheduling feature isn't just for convenience — it enables batch processing during off-peak hours, reducing rate-limit pressure and ensuring changes land before payroll cut-offs.

↩️ Selective Rollback

Our "Reverse Changes" feature in Edit History handles the product layer. At the infrastructure level, maintain a change log per system so rollbacks can target specific systems without affecting others that already succeeded.
